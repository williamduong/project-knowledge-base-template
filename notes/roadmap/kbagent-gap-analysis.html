§24 mismatch nghiêm trọng cần xử lý
Mismatch #1 — Dispatch Decision Tuple thiếu hoàn toàn
Repo dùng dispatch tuple (8 inputs) làm core routing mechanism cho mọi action. Mọi command, mọi gate, mọi pipe đều resolve qua tuple này. Documentation chúng ta hoàn toàn không có concept này — ta dùng "goal alignment check" rời rạc thay vì tuple deterministic.

Impact: blueprint hiện tại của ta không match runtime architecture của repo. User đọc xong sẽ confused khi gặp kbx dispatch command thực tế.

Status: closed by notes/roadmap/kbagent-dispatch-decision-tuple.html.
Mismatch #2 — Ontology entities khác hoàn toàn
Chúng ta design 8 node types (:Goal :Intent :Rule :KBDoc :Code :Signal :Release :Lesson) cho graph. Repo có 10 entities (Tenant · Subscription · ServicePrincipal · Project · Module · Config · Intent · Policy · CLICommand · Evidence) với Microsoft CDM mapping.

Chỉ overlap 1 entity: :Intent. Còn lại hoàn toàn khác. Đây là vấn đề lớn vì ontology là foundation.

Status: closed by notes/roadmap/kbagent-ontology-cdm-mapping.html (canonical mapping + SaaS extension rules + bi-temporal frontmatter model).
Mismatch #3 — Rule families và Principles
Chúng ta có 4 tier (AX/P/D/O) trong 1 hệ thống thống nhất. Repo tách thành 2 hệ thống: 25 Principles P0-P25 (governance) ở svfactory/principles.md + 8 rule families M/V/I/GB/AX/PR/WF/KA (runtime).

Lý do tách: Principles là human-readable governance directives (NL). Rules là machine-readable runtime checks. Đây là sự phân biệt cốt lõi mà ta chưa làm.

Status: closed in roadmap artifacts (complete-doc, roadmap, rules) with explicit layer separation wording.
Mismatch #4 — CLI naming + lifecycle states
Repo vừa đổi kb sang kbx ngày 2026-05-10 (commit 71d68f2). Toàn bộ examples trong documentation của ta dùng kb. Lifecycle states cũng khác: ta dùng backlog/active/staged/closed/archived, repo dùng DRAFT/PROPOSED/VERIFIED/EXECUTED/COMMITTED với state transition guards.

Mức độ: dễ fix nhưng cần search-and-replace toàn bộ documentation. Đây là breaking change naming gần nhất.

Status: naming and lifecycle wording normalized across roadmap docs and CLI command help/output references.

§4Final positioning — chúng ta đang ở đâu
Nhất quán theo nghĩa nào?
Documentation chúng ta đã build không sai về triết lý — 14/32 concepts identical với repo, 8/32 chỉ khác vocabulary, 6/32 là partial coverage. Đây là một simplified, SaaS-flavored view của hệ thống repo đầy đủ.

Nếu mục tiêu là documentation cho downstream SaaS user: documentation hiện tại OK với P0 fixes (naming + tách principles/rules).
Nếu mục tiêu là technical reference cho maintainer hoặc enterprise integrator: cần P1 (dispatch tuple + CDM ontology) + có thể P2 (gates + generative loop).
Nếu mục tiêu là roadmap cho việc build kế tiếp: 4 gap nghiêm trọng (Dispatch Tuple, CDM, bi-temporal, generative loop) chính là những thứ repo đã solve mà ta chưa biết — học từ repo trước, không cần re-design.
