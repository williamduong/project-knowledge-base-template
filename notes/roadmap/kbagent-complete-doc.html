Blueprint
KBAgent — evolvable, human-gated, copilot-driven · SaaS variant v2.7
evolvable3 layers10 flows28 rulesKuzu graph
KBAgent is evolvable, not self-evolving. Humans decide direction. System reduces friction, maintains evidence, and proposes — never acts unilaterally. SVFactory is legislative; KBAgent is executive.
3-layer architecture
Not self-evolving
Human approval at every gate. Copilot suggests, human approves, CLI executes. System proposes — never acts unilaterally.
Single graph write point
Graph written ONLY at ③ Retro via retro:write. All other ops read-only. Guarantees evidence quality and consistent state.
Domain-agnostic core
Primitives (Intent, Goal, Rule, Signal) contain zero domain logic. SaaS rules/goals live in config — swappable per domain.
Deterministic gate
kbx CLI exits 0 or 1 only. No partial success. No LLM in the gate path. Gate is predictable regardless of AI output quality.

Pipelines
4 main pipelines + 6 sub-pipelines. Main pipelines = human flows. Sub-pipelines = called internally.
P1 fixedP2 fixedP3 guidedP4 fixed
P1 — Intent lifecycle (8 steps)
Workflow lifecycle uses folder-based states like _backlog/, _active/, and _archive/. Ontology lifecycle uses formal states like DRAFT, PROPOSED, VERIFIED, EXECUTED, and COMMITTED; do not conflate the two.
1
createkbx intent create --title --type --repos[] --milestone
writes intent.md to _backlog/ · workflow state = backlog · ontology state = DRAFT
2
goal-alignkbx intent align — match intent vs active goals exit 0exit 1
exit 1: warn · human decides: keep in backlog workflow, convert to exception, or update goal
3
human-approvekbx intent approve — no auto-approve path
human reviews in Dashboard · explicit approval required (KBX-AX005)
4
activatekbx intent activate — enforces KBX-P001
checks: no other active intent · moves to _active/ · workflow state = active · ontology state = PROPOSED
5
execute ↺Copilot → human approve → kbx apply · sub: impact:check before each apply
repeats until staged · exit 1 on kbx apply → fix first · workflow state = staged · ontology state = EXECUTED
6
gatekbx gate — deterministic exit 0exit 1 · sub: doc:gate + ontology:validate + tests
exit 1 → back to step 5 with error report · no LLM in gate path (KBX-AX003) · ontology state must pass VERIFIED
7

evidence + :Lesson + :ADVANCES → criteria.current updates · workflow state = closed → _archive/ · ontology state = COMMITTED
8

triggers P2 if chaos spike · dashboard refreshes · next cycle begins or idles

P3 — Onboarding (6 steps)
1. init (detect new vs retrofit) → 2a. interview (Copilot Q&A) or 2b. scan source (scan-manifest.json) → 3. scaffold template + fill placeholders + seed goals/rules → 4. baseline chaos (first snapshot) → 5. kbx doctor exit 0 → 6. suggest 2-3 starter intents → handoff to P1.
P4 — Release (6 steps)
1. trigger (manual or N intents hit milestone) → 2. collect closed intents since last release → 3. release-gate (kbx gate --release, chaos ≤ 50, tests pass) → 4. version bump (semver) → 5. graph-write :Release + :INCLUDED_IN edges → 6. append release-catalog.json + export graph snapshot.

Rules — Domain & Operational
D = SaaS domain rules (replaceable per domain) · O = auto-generated by system over time
D rules — user can add/modifyO rules — auto-generated
Domain rules (D) are SaaS-specific. When you switch domain (research, e-commerce, enterprise), swap these rules via Flow 10. Operational rules (O) grow over time as the system learns repeated patterns from retrospectives.
Domain — Security SaaS · user can modify
KBX-S001
Auth middleware on all sensitive routes
Every route handling billing, PII, or admin actions must have auth middleware. Detected by chaos:scan route + auth pattern matching. Linked to INT-SAAS-001.
planned
KBX-S002
No secrets in source code
No API keys, tokens, or passwords as string literals. Detected by chaos:scan secret_patterns → blocks gate immediately. Process.env only.
planned
KBX-S003
Rate limiting on auth endpoints
Login, signup, password-reset, token-refresh must have rate limiting. Max 5 attempts/IP/min. Detected by checking middleware presence on auth route patterns.
draft
Domain — Billing SaaS · user can modify
KBX-B001
Billing provider is source of truth
Subscription status always fetched from billing provider (Stripe/Paddle). Cached state TTL ≤ 300s. Prevents stale subscription leading to access control errors. Linked to INT-SAAS-002.
planned
KBX-B002
Idempotent webhook handlers
All billing webhook handlers must be idempotent — processing same event twice produces same result. Implement using event ID deduplication store.
draft
Domain — Testing & Architecture SaaS · user can modify
KBX-T001
Integration tests on critical SaaS paths
Auth, billing, onboarding, and data export flows must have integration tests (not just unit tests). Detected by chaos:scan testing dimension checking critical path test coverage.
planned
KBX-D001
Multi-tenant data isolation
All DB queries on user data must include tenant_id scoping. No cross-tenant data leaks. Detected by scanning ORM/query patterns for missing where clauses on tenant-scoped models.
draft
KBX-D002
Graceful degradation on third-party failure
All third-party calls (billing, email, analytics, storage) must have timeout + fallback. No third-party failure causes a 500 to end user. Scan checks for missing try/catch + fallback.
draft
Operational — auto-generated by system grows over time
Operational rules are proposed by the system when it detects patterns. Human reviews and approves. After 3 months of use, expect 5–10 project-specific KBX-O rules.
KBX-O001
Lesson → rule promotion (3× same pattern)
When same-domain lesson appears in 3+ retrospectives, maintain loop proposes a draft :Rule. Human reviews Copilot-drafted body, edits, and approves. Applied via update:patch pipeline.
planned
KBX-O002
Chaos dimension spike → auto-suggest fix intent
When single chaos dimension increases by >8 points between consecutive snapshots, chaos:scan auto-proposes targeted fix intent. Appears in Dashboard signals. Human approves.
implemented
KBX-O003
High fan-in code → structural review intent
:Code node with fan_in > 30 triggers structural review intent proposal on next chaos:scan. High fan-in = systemic risk — any change affects 30+ dependents. Review checks interface stability.
planned
KBX-O004
Stale source-of-truth doc → review intent
source_of_truth=true :KBDoc with no linked intent :IMPLEMENTS for >90 days is flagged stale. Maintain loop proposes review intent. Prevents silent drift between docs and reality.
planned
Master rules
AX = constitutional (immutable) · P = process (adjustable via Flow 10)
AX — immutable, 7 rulesP — adjustable via Flow 10, 9 rules
Constitutional rules (AX) cannot be overridden by anyone, including the user. Violating them breaks system consistency. They are the axioms of KBAgent. Process rules (P) can be adjusted via Flow 10 (update:patch) with justification — but require NL normalization + human validation, not direct edits.
Constitutional — AX immutable
KBX-AX001
Separation of powers
SVFactory is legislative — produces templates and rules. KBAgent is executive — runs intents and maintains knowledge. These two must never mix. A KBAgent instance cannot modify its own template source. SVFactory cannot execute intents directly.
verified
KBX-AX002
Domain agnosticism of primitives
KBAgent core primitives (Intent, Goal, Rule, Signal, Evidence) must contain zero domain logic. Domain specifics live exclusively in ontology-contract.json and seed files. Engine code never hardcodes domain terms like "billing" or "auth".
verified
KBX-AX003
Deterministic gate — exit 0/1 only, no LLM in gate path
kbx CLI exits exactly 0 (pass) or 1 (fail). No partial success, no warnings-as-pass, no LLM judgment in the gate path. If any check fails, the entire gate fails. Human reviews exit 1 — no auto-retry, no auto-fix.
verified
KBX-AX004
Single graph write point — retrospective only
The Unified Knowledge Graph may only be written during the retrospective step via retro:write sub-pipeline. All other operations (chaos scan, intent activation, goal check) are read-only on the graph. Enforced at Kuzu connection level.
verified
KBX-AX005
Human approval at every intent gate
No intent — exec, fix, or exception — may advance from draft to active without explicit human approval via kbx intent approve. Signal detector proposals, maintain fix intents, and goal-aligned suggestions all require explicit approval. There is no auto-approve path.
verified
KBX-AX006
Evidence on every node and edge — no anonymous data
Every node and every edge written to the Unified Graph must carry evidence metadata: source, confidence, timestamp, linked_intent_id, version, repo_origin. Writes without evidence are rejected by retro:write. No anonymous knowledge.
verified
KBX-AX007
Goals never auto-close — human changes only
Goals are invariants. No pipeline, chaos spike, or criteria threshold crossing may automatically close or supersede a goal. Goals move to warning or superseded only via explicit human action. System emits warnings; it may not act on them autonomously.
verified
Process — P adjustable via Flow 10
KBX-P001
One active intent per version branch
Only one intent may be active|staged per branch. A second activation requires the first to be closed or archived. Exception intents may coexist with --allow-parallel flag (logged). Enforced by kbx intent activate checking _active/ count.
verified
KBX-P002
Three-layer vibe execution
All work: NL input → CLI normalize → CLI execute → AI completion. Copilot suggests before the gate. Copilot summarizes after the gate. Copilot never executes. Dashboard suggest buttons copy prompts — they never call kbx directly.
verified
KBX-P003
Goal alignment required before activation
Every exec intent must pass goal-align check. Alignment means the intent's work will advance at least one criterion of at least one active goal. Fix and exception intents are exempt with explicit type declaration and human approval.
verified
KBX-P004
Impact check before each kbx apply
impact:check sub-pipeline must traverse :AFFECTS/:DEPENDS_ON/:BREAKS before every kbx apply. Result shown to human before confirmation. Cannot be skipped without --skip-impact flag (which is logged as a :Signal node).
implemented
KBX-P005
Maintain-first when chaos ≥ 70 (unstable)
When chaos score = unstable|chaotic, no new exec intent may activate until score drops below 60. System enters maintain-first mode. Fix intents take priority. Enforced by P1:activate checking latest chaos snapshot level.
implemented
KBX-P006
Retrospective is mandatory — no skip
An intent may not be archived without completing a retrospective. kbx intent archive without prior retro exits 1. Retro must include evidence note and one lesson candidate. kbx intent archive checks retro_completed=true before proceeding.
verified
KBX-P007
Release gate requires chaos ≤ 50
A release (P4) may not proceed if latest chaos snapshot is above 50. Gate checks chaos as part of exit 0/1 decision. Hotfix releases may bypass with --hotfix flag, which is logged as a :Signal node for audit.
planned
KBX-P008
doc:gate blocks commit if coverage below threshold
doc:gate checks: KB doc coverage %, placeholder ratio (<15% threshold), and valid frontmatter on all .md files. Fails gate with exit 1 if any check fails. Called by P1:gate and P4:release-gate.
implemented
KBX-P009
NL patch must normalize before apply
Any change to natural language content (rules, principles, KB docs, goal criteria descriptions) must go through update:patch sub-pipeline: NL → canonical → meaning diff → human validates diff → apply as intent. Direct text edits without this pipeline are flagged by ontology:validate.
planned
