Workflow lifecycle is separate from ontology lifecycle: workflow states are folder-based (backlog, active, staged, closed, archived), while ontology states are formal (DRAFT, PROPOSED, VERIFIED, EXECUTED, COMMITTED). Governance Principles live separately in svfactory/principles.md as P0-P25; AX/P below are runtime rules.

KBX-AX001 Separation of powers governance verified

SVFactory is legislative — produces templates and rules. KBAgent is executive — runs intents and maintains knowledge. These two must never mix. A KBAgent instance cannot modify its own template source. SVFactory cannot execute intents directly.

RationalePrevents circular dependency between the system that produces agents and the agents themselves. Ensures upgrades are deliberate (Flow 10), not accidental.
Enforced bykbx gate rejects commits that modify .kb/graph/ outside retro step

KBX-AX002 Domain agnosticism of primitives governance verified

KBAgent core primitives — Intent, Goal, Rule, Signal, Evidence — must contain zero domain logic. Domain specifics (repo_origin enum, SaaS goal templates, security rules) live exclusively in ontology-contract.json and seed files. The engine code never hardcodes "billing" or "auth".

RationaleEnables KBAgent to work for any domain (SaaS, research, blog, enterprise) by swapping the contract, not the engine.
Enforced byontology:validate sub-pipeline · PR review on engine changes

KBX-AX003 Deterministic gate — no LLM in the gate path execution verified

kbx CLI exits exactly 0 (pass) or 1 (fail). No partial success, no warnings-as-pass, no LLM judgment in the gate path. If a check fails, the entire gate fails. Human reviews exit 1 — no auto-retry, no auto-fix.

RationaleNon-deterministic gates produce flaky pipelines. Trust requires predictability. Copilot assists before the gate, never inside it.
Enforced bykbx gate process.exit(0|1) — no other exit codes allowed
P1:gateP4:release-gate
KBX-AX004 Single graph write point — retrospective only data verified

The Unified Knowledge Graph may only be written during the retrospective step (retro:write sub-pipeline). All other operations — chaos scan, intent activation, goal check — are read-only on the graph. This is the single most important data integrity rule.

RationalePrevents inconsistent state from concurrent or partial writes. Evidence quality is guaranteed when writes are batched at retro. Graph state always reflects completed, reviewed work.
Enforced byKuzu connection opened read-only except during retro:write call · write guard in engine layer

KBX-AX005 Human approval at every intent gate governance verified

No intent — exec, fix, or exception — may advance from draft to active without explicit human approval. Signal detector proposals, maintain fix intents, and even goal-aligned suggestions all require kbx intent approve. There is no auto-approve path.

RationaleThis system is evolvable, not self-evolving. Human agency is the fundamental guarantee. Removing this creates autonomous drift.

KBX-AX007 Goals never auto-close — human changes only governance verified

Goals are invariants. No pipeline, no chaos spike, no criteria threshold crossing may automatically close or supersede a goal. Goals move to warning or superseded only via explicit human action (kbx goal update). The system may emit warnings; it may not act on them autonomously.

RationaleGoals are the North Star. Auto-closing them when criteria are met removes the long-term compass. A goal achieved usually spawns a higher goal — that is a human decision.
Enforced byGoal engine: status transitions require explicit command · kbx goal close requires --confirm flag

KBX-P001 One active intent per version branch intent verified

Only one intent may be in active or staged workflow state per version branch at any time. These are workflow lifecycle states, not ontology states. A second activation requires the first to be closed or archived. Exception intents pending approval do not count toward this limit.

RationaleParallel active intents create merge conflicts, unclear impact attribution, and diffuse focus. One thing at a time per branch.
Enforced bykbx intent activate checks _active/ count · rejects if count > 0
Overrideexception type intent can coexist · requires explicit --allow-parallel flag with justification
P1:activate
KBX-P002 Three-layer vibe execution execution verified

All work follows: NL input (human or Copilot) → CLI normalize (structured command) → CLI execute (deterministic) → AI completion (post-gate). Copilot may suggest before the gate. Copilot may summarize after the gate. Copilot never executes.

RationaleKeeps execution auditable and reversible. AI-generated code is always reviewed by human before CLI commit. Gate is deterministic regardless of AI quality.
Enforced byDashboard: suggest buttons copy prompts, they never call kbx directly · kbx apply requires human confirmation

KBX-P003 Goal alignment check before activation intent verified

RationalePrevents "busy work" that doesn't advance any goal. Forces intentional connection between tasks and strategy.

KBX-P004 Impact check before each apply data implemented

Before every kbx apply in the execute loop, impact:check must traverse the graph for :AFFECTS, :DEPENDS_ON, and :BREAKS edges from the files being changed. The result is shown to human before confirmation. Human may proceed or abort.

KBX-P005 Maintain loop triggered by chaos threshold chaos implemented

When chaos score reaches unstable (≥70), the maintain loop must be triggered before any new exec intent can be activated. The system enters "maintain-first mode" until score drops below 60. Fix intents take priority over exec intents.

RationaleBuilding on an unstable foundation compounds debt. Forcing maintain-first prevents the chaos score from becoming a vanity metric.
Enforced byP1:activate checks latest chaos snapshot · blocks if level=unstable|chaotic and no active fix intent

KBX-P006 Retrospective is mandatory — no skip data verified

An intent may not be archived without completing a retrospective. kbx intent archive without prior retro exits 1. The retro must include at minimum: evidence note and one lesson candidate (even if empty string is not accepted — requires actual content).

RationaleRetrospectives are the learning mechanism. Skipping them means the graph never grows, criteria never update, and the system never improves.
Enforced bykbx intent archive checks retro_completed=true in intent.md · rejects if false

KBX-P007 Release gate requires chaos ≤ 50 chaos planned

A release (P4) may not proceed if the latest chaos snapshot is above 50 (manageable threshold). The release gate checks chaos score as part of exit 0/1 decision. Exception: hotfix releases may bypass with --hotfix flag, which is logged as a :Signal node.

RationaleReleasing with high chaos ships technical debt to production. The threshold of 50 is deliberately lower than the unstable threshold (70) — releases demand better hygiene than development.

KBX-P009 NL patch must normalize before apply execution planned

Any change to natural language content (rules, principles, KB docs, goal criteria descriptions) must go through the update:patch sub-pipeline: NL → canonical form → meaning diff → human validates diff → apply as intent. Direct edits to rule body text without this pipeline are flagged by ontology:validate.

RationaleNL changes are the hardest to audit. Subtle wording shifts can change system behavior. Normalizing to canonical form makes diffs reviewable.

KBX-B001 Billing provider is source of truth billing planned

Subscription status, plan limits, and payment state must always be fetched from the billing provider (Stripe, Paddle, etc.) and never cached without TTL. Locally cached subscription state must have TTL ≤ 300s. Detected by scanning for subscription/plan reads without TTL patterns.
