Get started

Getting Started In 3 Steps

Before you begin, make sure you have Node.js 18+, Git, and a code editor (VS Code, Cursor, or Claude recommended). No other runtime or service is required to bootstrap a local KB. Read the full setup guide in Docs →

01

Install + Initialize KB

One-liner with npx (no global install) is the fastest path. Or install globally if you'll use kb often. Mode is auto-detected (private-git if .git exists, otherwise tracked). The @kb agent and /kb-* prompts only appear after kb init writes them into your repo.

Terminal — fastest (no install)
cd <your-repo>
npx @williamduong/kb@latest init --yes
Terminal — global install
npm install -g @williamduong/kb@latest
cd <your-repo>
kb init
Setup guide Release notes
02

Drive the KB from chat with /kb-run

Open Copilot Chat (VS Code, Cursor, Claude) in your workspace and run /kb-run. It auto-inits if needed, writes a runtime plan, and executes one resumable step at a time.

Copilot Chat (VS Code / Cursor / Claude)
/kb-plan      # analyze workspace, write knowledge-base/.kb/runtime-plan.md
/kb-run       # execute next pending step (resumable)
/kb-ask       # read-only KB-first Q&A (no edits)

@kb What database does this project use?
@kb What are the main components?
@kb How do I add an extension?
@kb audit metadata
@kb status
Agent guide Code Q&A index
03

Review & Publish

Review generated stubs, fill in content, then run kb maintain for one-command sync and validation. Your KB is now ready for team collaboration and AI agent reference.

Terminal
kb maintain

# (Optional) faster checks for large repositories
kb maintain --fast

# (Optional) reconcile baseline/version metadata
kb update

CLI Backbone + Agent Operating Model

The CLI manages the KB lifecycle. The agent stays disciplined because the KB already contains the operating manual, prompt pack, and governance rules it must read first.

CLI Backbone

Scaffold, fill, track, and maintain — the full KB lifecycle in one CLI.

Scaffold & Fill

kb init Scaffold KB; auto-detect mode, create KB agent prompts, and generate the relevant AI IDE adapter
kb bootstrap Scan source code; generate stub docs for arch, API, DB, and ops tiers
kb index Build fill-rate summary: doc count, placeholder count per tier
kb questions Generate intake questions from placeholders; --chat --batch N for 5-question AI chat batches
kb mark Set kb_state on any doc: template → autofilled → needs-review → verified
kb normalize-state Assign kb_state: template to all docs that are currently unset

Lifecycle

kb plan Manage finalization plan items (list / add)
kb show / hide Expose or retract the KB mount in private-git mode
kb status Report KB presence, mode, and state with a --json mode for prompts and CI
kb ide enable / disable IDE integration (Copilot, Cursor, Claude) without touching internals
kb maintain One-command maintenance pipeline: sync + doc checks + test + doctor
kb sync Collect drift evidence since baseline; write sync report
kb update Run sync then stamp the new template version state (refuses to downgrade)
kb doctor Publish-readiness checks; --strict exits 1 on any WARN
kb test Smoke-test template structure against expected schema
kb uninstall Remove KB content and generated AI helper files; --keep-ai-files preserves IDE adapters
Agent Operating Model

The KB is the agent’s source of truth. The CLI keeps it coherent; the agent reads first, then acts.

  • Read before acting Opens INDEX.md + operating manual on every session start.
  • Prompt Pack Pre-built workflows for each action — no ad-hoc processes invented.
  • Revision-aware Checks baseline stamp to detect stale docs before trusting content.
  • Verify after writing Runs kb doctor and prepares KB changes to ship with related code updates.

The KB fits into three recurring moments in any project lifecycle.

Step 2 — KB runs automatically

3 Source sync — reflecting code changes back into KB

After a successful bootstrap, run kb maintain as your default maintenance path. Use kb sync --accept-baseline only after reviewing drift evidence.

  • Updates affected KB docs to match source
  • Stamps revision state + publishes sync report
Step 3 — Continue execution Review the sync report, then continue your current coding or documentation workflow with refreshed KB context.

What’s coming

Roadmap & Planned Features

The KB template ships in versioned waves. Each wave stands alone in user value while building the foundation for the next. Dates are not committed; ordering is.

v1.3 — Git binding & impact surface (shipped)

Bind any KB doc to one or more code paths via kb bind. kb scan walks the diff since the last baseline and reports which docs are impacted, with KB self-edits filtered out.

v1.4 — Impact engine, recursive expansion, cycle detection (shipped)

An in-memory graph (graphology) turns a code change into a ranked list of affected docs. kb scan --recursive expands related_strong neighbours into a transitive impact list, kb verify stamps a verified commit per doc, kb baseline manages the source git baseline, and kb doctor flags strong-edge cycles. Governance: 15-governance/impact-policy.md.

v1.5 — Catalog & deterministic release notes

A queryable catalog of every doc, binding, and verification stamp. Release notes are generated deterministically from git history and impact data, ready for any downstream KB.

v1.6 — Release pipeline orchestrator (optional)

Skippable wave. Adds an opinionated tag → notes → publish pipeline behind a decision gate, for teams who want one command to ship rather than a checklist.

v2.0 — Intent layer & KB Agent

A 7-step intent workflow that observes user actions, captures lessons learned, and feeds them back into the KB. The bundled KB Agent enforces the workflow. Major version — includes a migration path.

v3.0 — 3-tier KB platform (draft, gated)

Workspace, organisation, and platform tiers with shared identity and cross-repo discovery. Locked behind explicit trigger conditions; not committed until usage from v2.x justifies the lift.
